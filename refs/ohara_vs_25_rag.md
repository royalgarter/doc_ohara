# Ohara vs 25 RAG Types

| Type | Definition | Assessment |
|---|---|---|
| **Standard RAG** | Fetches relevant info chunks from a knowledge base and merges them into a prompt for the LLM to synthesize a clear answer. The baseline retrieve-then-read pattern. | **Covered.** BM25 Phase 1 fetches and ranks chunks; Phase 4 fusion merges into single ranked result set. |
| **Conversational RAG** | Leverages full dialogue history so each retrieval query is conditioned on prior turns, enabling context-aware follow-up without re-stating prior context. | **Gap — Medium priority.** No session state; every `query()` call is cold. **What:** carry prior Q&A across turns. **Why:** anaphora breaks retrieval ("what about *its* governance?" resolves to nothing). **How:** accept `sessionHistory[]` param in `query()`; prepend prior turns to query fingerprint prompt so Gemini resolves references before Phase 1. |
| **Corrective RAG** | After retrieval, a critic model (or scoring step) evaluates chunk relevance and either fixes, filters, or re-retrieves to replace low-quality results before generation. | **Gap — High priority.** Phase 3 structural traversal follows HAS_CHILD/NEXT_SIBLING blindly regardless of topical fit, injecting noise. **What:** post-Phase-3 relevance filter before fusion. **Why:** structurally-proximate ≠ semantically relevant. **How:** compare each Phase 3 node's `sumo_tags` against query `sumo_hints`; drop nodes below overlap threshold before fusion. |
| **Hybrid RAG** | Combines sparse keyword search (BM25/TF-IDF) with dense semantic search (embeddings) and merges results via score fusion or re-ranking for broader coverage. | **Covered.** BM25 full-text (Phase 1) + SUMO ontology tag expansion (Phase 1b) = keyword + semantic hybrid. |
| **Speculative RAG** | Anticipates likely follow-up queries and preloads relevant documents in advance, reducing latency by pre-warming the retrieval cache before the user asks. | **Gap — Low-Medium priority.** No pre-warming. **What:** background-trigger `query()` on Explorer frontier after Principal tier is returned. **Why:** next user question often follows Explorer candidates. **How:** fire async `query()` calls on top frontier nodes after main response; cache with short TTL keyed on frontier node IDs. |
| **Memory-Augmented RAG** | Maintains a persistent memory store (episodic, semantic, or working) alongside the retrieval index so prior interactions or domain facts enrich current context. | **Covered.** Entity graph nodes (`entities` collection) persist canonical named-entity knowledge with `aliases`, `mention_count`, `document_ids` across all ingested docs. |
| **Fusion RAG** | Runs multiple retrieval strategies in parallel (different queries, indexes, or methods) and combines all result lists into a single merged response via RRF or weighted fusion. | **Covered.** Phase 4 weighted sum: BM25×1.0 + SUMO×0.4 + entity×0.6 + cross-doc×0.4 + structural×0.3. Dedup by `_id`, sort descending. |
| **Context-Aware RAG** | Uses request metadata — user intent, session state, document type, temporal signals — to filter or re-rank retrieval results before passing them to the LLM. | **Covered.** Phase 0a query fingerprint (Gemini) extracts `sumo_tags`, `entity hints`, `temporal_intent` from raw query. Gates temporal scoring, SUMO expansion, entity pivot per query intent. |
| **Agentic RAG** | An autonomous agent controls the retrieval loop — deciding which tools to call, when to re-query, and when enough evidence has been gathered — rather than a fixed pipeline. | **Partially covered.** Explorer tier identifies frontier candidates but has no autonomous agent loop. Caller/UI decides whether to pursue frontier. Full agentic loop (agent decides when to re-query) not wired. |
| **RL-RAG** | Applies reinforcement learning (reward = answer quality signal) to train a policy that optimizes retrieval strategy: what to query, which sources to weight, when to stop. | **Skip.** Requires large interaction corpus. Phase weights (`OHARA_ENTITY_PIVOT_WEIGHT` etc.) are already tunable env vars. REFEED RAG covers the feedback angle with far less infra. |
| **Self-RAG** | The model generates, then critiques its own output — inserting special reflection tokens to decide whether retrieval was needed, whether results were relevant, and whether the answer is supported. | **Gap — Medium priority.** No self-reflection on answer quality before returning results. **What:** model checks whether retrieved chunks actually answer the query. **Why:** high-scoring nodes can be topically adjacent but non-responsive. **How:** optional Gemini one-shot pass at fusion time (same pattern as `OHARA_INTEGRITY_LLM_VERIFY`) to flag and filter non-responsive nodes before final sort. |
| **Sparse RAG** | Extracts only the most essential, highest-signal information from retrieved chunks, discarding verbose or redundant content for concise, focused output. | **Skip.** Artifact filter at ingest (step ⑦) already removes separator lines, TOC noise, and short nodes. Fragment reattachment handles orphaned content. Problem solved at source. |
| **Adaptive RAG** | Dynamically adjusts retrieval scope, depth, and strategy based on query complexity — simple queries get shallow retrieval; complex queries trigger multi-step or agentic retrieval. | **Covered.** Query classified as keyword (≤3 tokens) / phrase (4-30) / paragraph (>30); classification gates Phase 0a fingerprinting, SUMO expansion, and temporal scoring. Scope adapts to complexity. |
| **Citation-Aware RAG** | Tracks and surfaces the provenance of each claim in the generated answer — linking output passages back to specific source documents, pages, or chunks with verifiable citations. | **Covered.** Every result carries `edge_verb`, `edge_summary`, `hops` (cross-doc path), and `provenance[]` array (phase + document + edge metadata) for full source traceability. |
| **REFEED RAG** | Uses explicit user feedback (thumbs up/down, edits, re-queries) iteratively to improve retrieval — adjusting weights, re-ranking, or updating the index based on observed quality signals. | **Gap — Low priority (requires UI).** Phase weights are static env vars; no feedback loop. **What:** user feedback ("not helpful") propagates to adjust retrieval weights. **Why:** optimal phase weights vary by corpus. **How:** log `(query, result_id, feedback)` tuples; periodic job tunes `OHARA_*_WEIGHT` vars via bandit or gradient update. |
| **Multimodal RAG** | Retrieves and integrates information across modalities — text, images, audio, tables, diagrams — using modality-specific encoders and a unified retrieval index. | **Skip (defer).** Figure nodes exist in DoCO schema. Image embeddings require separate vision pipeline and vector infra. Structural graph already handles document layout. Revisit when `OHARA_EMBED_PARAGRAPHS` vector path matures. |
| **Multi-Hop RAG** | Chains multiple retrieval steps sequentially — each retrieved document informs the next query — enabling the system to connect facts spread across several sources. | **Covered.** Phase 1c follows `SIMILAR_TO` edges up to `OHARA_CROSS_DOC_EXPAND_DEPTH` hops (graph traversal `1..expandDepth`). Result carries `hops` count from seed document. |
| **Reasoning RAG** | Couples retrieval with explicit reasoning tools (chain-of-thought, scratchpads, symbolic solvers) so the model can derive answers that require inference, not just lookup. | **Partially covered.** `OHARA_INTEGRITY_LLM_VERIFY` hooks a one-shot Gemini cross-check for Integrity-tier candidates. No chain-of-thought between retrieval phases. Full reasoning loop (retrieve → reason → retrieve) not implemented. |
| **Long-Context RAG** | Handles retrieval over very long documents or large corpora by using extended context windows to accommodate full documents or large retrieved passages without truncation. | **Skip.** DoCO chunking + structural graph *is* the architectural answer: long documents split into typed nodes (Chapter/Section/Paragraph) and re-joined at query time via structural traversal. No extended context window needed. |
| **Federated RAG** | Searches across multiple distributed databases, APIs, or knowledge bases simultaneously, federating results from heterogeneous sources into a unified response. | **Skip.** ArangoDB is the single source of truth by design. Web crawl ingest bundles external sources into the same graph. Distributed DB federation adds ops complexity with no clear benefit given current architecture. |
| **Hierarchical RAG** | Organizes retrieval in a coarse-to-fine hierarchy — first retrieving high-level summaries or headings, then drilling into relevant sections for detail. | **Covered.** Structural traversal (Phase 3) navigates HAS_CHILD/NEXT_SIBLING/BELONGS_TO hierarchy. Tier system (Principal → Integrity → Explorer) implements coarse-to-fine ranking over the result set. |
| **Context-Ranking RAG** | Re-ranks retrieved chunks by contextual importance relative to the query — using cross-encoders, LLM judges, or learned rankers — before passing to the generator. | **Covered.** Phase 4 fusion scores by weighted sum; tier classification adds percentile floor (`OHARA_PRINCIPAL_SCORE_PCTL`) and multi-doc diversity requirement. `provenance[]` explains what corroborated each entry. |
| **Prompt-Augmented RAG** | Converts raw retrieved data into structured prompt inputs — reformatting, summarizing, or templating chunks — so the LLM receives clean, well-organized context rather than raw text. | **Covered.** Ingest converts raw Markdown to typed DoCO nodes (Chapter/Section/Paragraph/Table) via Gemini. Query side uses structured fingerprint prompt to convert raw text to `sumo_tags + entity hints` before retrieval. |
| **Few-Shot RAG** | Retrieves examples that match the query pattern and includes them as few-shot demonstrations in the prompt, aligning generation style and format to the user's intent. | **Skip.** Gemini prompts (`prompts/ingest_document.md`, `prompts/extract_query_fingerprint.md`) already embed typed examples inline. No separate few-shot retrieval architecture needed. |
| **Chain-of-Retrieval (CoR)** | Performs sequential retrieval loops where each iteration refines the query based on what was retrieved — progressively narrowing toward a precise answer across multiple rounds. | **Gap — High priority.** Explorer tier identifies frontier candidates but retrieval is one-shot; frontier is never chased. **What:** iterative loop where each retrieval feeds the next query. **Why:** deep multi-hop knowledge (e.g., influence chains across 3+ docs) unreachable in single pass. **How:** wrap `query()` in a CoR loop — use top Explorer `edge_verb`/`edge_summary` as augmented query seeds for the next iteration; stop at N iterations or score plateau; merge all iteration results before final fusion. |
